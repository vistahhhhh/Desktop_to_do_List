# 置顶逻辑问题梳理（先不改代码）

## 1. 当前现象（来自你的反馈）
- 主窗口（桌面待办卡片）置顶后，拖出的便签窗口有时会被主窗口覆盖到下面。
- 你期望：**拖出的便签默认显示在主窗口之上**，并且在使用中尽量“符合直觉”。

## 2. 参与对象与概念
- **主窗口置顶**：主程序的卡片窗口（MainWindow）通过 `Qt.WindowStaysOnTopHint` 控制。
- **浮动便签置顶**：每个 `FloatingNoteWindow` 自己也可以通过 `Qt.WindowStaysOnTopHint` 控制。
- **窗口层级（Z-order）**：同样是置顶窗口之间也存在前后顺序；并且不同的 flag（如 `Qt.Tool`）在 Windows 上也可能影响层级表现。

## 3. 为什么会出现“被覆盖/压到下面”
在 Windows 下，常见触发点：
- **主窗口是置顶**，而浮动便签是非置顶：主窗口天然压住它。
- 两者都置顶时：
  - 谁最后一次 `raise_()` / `activateWindow()`，谁更可能在上面。
  - 修改 `setWindowFlags()` 后通常需要 `show()`，会导致窗口重新进入层级队列。
- `Qt.Tool` 窗口与普通窗口在任务切换/焦点变化时层级表现会更“跳”。

## 4. 你期望的“直觉规则”（建议作为后续实现目标）
这里我先把规则写清楚，等你确认后再改代码。

### 4.1 默认规则
- **R1：拖出便签 = 浮动便签默认不置顶，但默认在主窗口之上**
  - 理由：不希望它在切换到其它应用后仍然“霸占最上层”，但拖出时需要立刻可见且在主窗口上方。

- **R2：浮动便签的置顶只影响自己**
  - 你之前明确说过：置顶按钮只控制该便签。

### 4.2 与主窗口置顶的交互规则（关键）
- **R3：当主窗口置顶打开时**
  - 浮动便签默认不置顶（R1），此时主窗口会天然压住它。
  - 因此拖出时需要额外保证“刚拖出那一刻”浮动便签显示在主窗口之上：
    - 刚创建/显示浮动便签后，调用 `raise_()` + `activateWindow()`。

- **R4：当主窗口取消置顶时**
  - 浮动便签若仍置顶，应保持置顶（不被主窗口状态影响）。

### 4.3 关闭/收回的边界规则
- **R5：浮动便签关闭/收回时的主面板显示**
  - 更符合直觉的两种可能：
    - 显示刚关闭的便签（你之前反馈希望这样）。
    - 或者回到新建空白（若刚关闭的便签已被删除/无内容）。

## 5. 需要重点验证的场景清单（建议你按这个顺序体验）
- **S1**：主窗口置顶=开 -> 拖出便签 -> 浮动便签应在最上层。
- **S2**：主窗口置顶=开 -> 同时打开多个浮动便签 -> 新打开/刚操作的那一个应在最上层。
- **S3**：主窗口置顶=关 -> 浮动便签置顶=关（默认）-> 拖出后默认在主窗口之上；切换到其它应用再回来，不要求它仍在最上。
- **S4**：浮动便签取消置顶 -> 主窗口置顶=开 -> 浮动便签应被主窗口压住（此为“符合规则”的结果）。
- **S5**：拖出的是空白便签 -> 不输入内容直接关闭 -> 不应留下空便签记录。

## 6. 后续代码改动方向（先列出，不动代码）
- 拖出创建 `FloatingNoteWindow` 后：
  - 默认不加 `Qt.WindowStaysOnTopHint`。
  - `win.raise_()` + `win.activateWindow()`，确保拖出后默认在主窗口之上。
- 对“置顶按钮”的视觉状态保持一致（已完成视觉反馈部分）。
