# 任务-便签关联从多对多改为一对一：改造方案（评审稿）

## 1. 目标

将当前“任务 ↔ 便签”**多对多**改为**严格一对一（0..1 ↔ 0..1）**：

- 一个任务最多关联一个便签
- 一个便签最多关联一个任务
- 保留“无关联”状态（可空）

并且：

- 兼容已有数据（做可追溯迁移）
- UI 语义从“关联集合”改为“关联对象”
- 服务层 API 从列表/批量改为单对象操作

---

## 2. 现状（受影响点）

当前实现核心依赖中间表 `task_note_links`：

- 模型：`src/models/task_note_link.py`
- 服务：`src/services/link_service.py`
- 任务计数：`src/services/task_service.py` 中 `get_task_link_count`
- 便签创建关联：`src/services/note_service.py` 中 `create_for_task`
- UI 弹窗：`src/ui/link_dialog.py`（查看关联便签/关联已有/便签侧关联任务）
- 主流程：`src/ui/main_window.py`（关联入口、批量关联、查看关联）
- 测试：`tests/test_phase8_task_note_linking.py`

## 3. 目标数据模型（建议）

### 3.1 设计选择

采用“**在 notes 表上增加 task_id**”方案：

- `notes.task_id`：`INTEGER NULL UNIQUE`
- 外键：`FOREIGN KEY(task_id) REFERENCES tasks(id) ON DELETE SET NULL`

这样可满足：

- 便签可独立存在（`task_id` 为 NULL）
- 删除任务不删除便签，只解除关联（与你当前业务一致）
- 一对一由 `UNIQUE(task_id)` 保证

### 3.2 模型关系

- `Task.linked_note`：`relationship("Note", uselist=False, back_populates="linked_task")`
- `Note.linked_task`：`relationship("Task", uselist=False, back_populates="linked_note")`
- 删除 `TaskNoteLink` 模型及 `Task.note_links` / `Note.task_links`

## 4. 数据迁移方案（重点）

## 4.1 迁移原则

旧数据可能存在多对多冲突，迁移必须可解释、可审计。

### 4.2 冲突解决规则（建议固定）

对每个 task 仅保留 1 条 note，按以下优先级：

1. note `updated_at` 最新
2. 若相同，`task_note_links.created_at` 最新
3. 若仍相同，note.id 最大

并且 note 也只能归属一个 task：

- 若同一 note 被多个 task 选中，只保留“优先级最高”的那个 task（同上规则）
- 被淘汰关系写入备份表 `task_note_link_migration_backup`

### 4.3 迁移步骤

1. `ALTER TABLE notes ADD COLUMN task_id INTEGER NULL`
2. 根据旧 `task_note_links` 计算保留映射并回填 `notes.task_id`
3. 创建唯一索引 `UNIQUE(notes.task_id)`
4. 创建普通索引 `INDEX(notes.task_id)`
5. 写入 backup 表（保留被淘汰关系）
6. 代码切换完成并验证后，删除 `task_note_links`（可延迟到下一版本）

## 5. 服务层改造

## 5.1 LinkService 重写为 OneToOneLinkService（可沿用类名）

建议 API：

- `bind_note_to_task(task_id, note_id) -> bool`
  - 若 note 已绑定其他 task：先拒绝（或按策略转绑，需配置）
- `unbind_by_task(task_id) -> bool`
- `unbind_by_note(note_id) -> bool`
- `get_note_for_task(task_id) -> Note | None`
- `get_task_for_note(note_id) -> Task | None`
- `has_note(task_id) -> bool`

删除 API：

- `get_notes_for_task`（列表）
- `batch_link`
- `count_notes_for_task`（恒为 0/1，可保留兼容层）

## 5.2 NoteService

- `create_for_task(task_id)`：创建 note 后直接写 `note.task_id = task_id`
- 保存/删除逻辑不变

## 5.3 TaskService

- `get_task_link_count` 返回 0 或 1
- 任务删除/永久删除时不再清理 link 表，改为依赖 `ON DELETE SET NULL` 或显式置空
